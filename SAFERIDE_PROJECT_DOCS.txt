================================================================================
  1. PROJECT OVERVIEW
================================================================================

SafeRide is a full-stack, multi-service school transport safety platform. It
provides real-time GPS tracking, face-recognition-based student boarding
verification, instant parent alerts, and a centralised admin dashboard for
fleet management.

Original codename internally: GuardianSync v2.0

  7. MODULE 4: API GATEWAY (BACKEND)
================================================================================

  Location:  e:\SafeRide-master\api-gateway\
  Port:      3000
  Run:       npm run dev (from api-gateway/)
  Main:      src/index.js

  ENTRY POINT
  ───────────
  src/index.js
    - Bootstraps the Express app.
    - Connects to MongoDB (mongoose).
    - Registers all middleware, routes, socket handlers.
    - Starts HTTP server on port 3000.

  MODELS (src/models/)
  ────────────────────
  User.model.js
    - Schema for both parents and admins.
    - Fields: email, passwordHash, role ("parent"/"admin"), name, phone,
      children[] (parent), notifications[], createdAt.
    - bcrypt hashing on pre-save hook.

  Bus.model.js
    - Schema for each bus in the fleet.
    - Fields: registrationNumber, capacity, assignedRoute, driverRef,
      currentTripId, status.

  BusLocation.model.js
    - Schema for GPS location records.
    - Fields: busId, latitude, longitude, speed, heading, timestamp,
      tripId, studentsOnBoard[].
    - TTL index: records auto-delete after 90 days.

  Student.model.js
    - Schema for students.
    - Fields: name, parentRef, busRef, faceEncoding (128-float array),
      faceImageUrl, boardingHistory[], isPresent, rfidTag.

  CONTROLLERS (src/controllers/)
  ──────────────────────────────
  auth.controller.js
    - POST /api/auth/register   - Create parent or admin account.
    - POST /api/auth/login      - Validate credentials, return JWT.
    - POST /api/auth/refresh    - Refresh expired JWT.
    - POST /api/auth/logout     - Invalidate token.

  student.controller.js
    - GET/POST/PUT/DELETE on /api/students.
    - On student photo upload: forwards image to AI service
      (POST http://ai-service:8001/api/v1/face/encode).
    - Stores returned 128-float encoding in Student.model.

  scan.controller.js
    - POST /api/scan/verify     - Called by driver app when scanning a face.
    - Fetches all students' encodings for the bus from DB.
    - Forwards probe encoding + known encodings to AI service
      (POST http://ai-service:8001/api/v1/face/identify).
    - If match found: marks student as boarded, fires Socket.IO event,
      triggers parent notification.

  ROUTES (src/routes/)
  ────────────────────
  auth.routes.js          - /api/auth/* endpoints
  bus.routes.js           - /api/buses/* (CRUD for fleet)
  parent.routes.js        - /api/parent/* (dashboard, children, history)
  scan.routes.js          - /api/scan/* (face scan verification)
  student.routes.js       - /api/students/* (CRUD + photo upload)
  user.routes.js          - /api/users/* (profile, settings)

  MIDDLEWARE (src/middleware/)
  ───────────────────────────
  auth.js
    - JWT verification middleware.
    - Decorates req.user with decoded token payload.
    - Enforces role-based access (admin vs parent routes).

  validate.js
    - Zod schema validation middleware for request body/params.

  errorHandler.js
    - Global Express error handler.
    - Formats all errors into consistent JSON responses.

  SERVICES (src/services/)
  ────────────────────────
  notificationService.js
    - Sends email via Nodemailer (SMTP).
    - Sends SMS via Twilio API.
    - Saves notification record to parent's User document.
    - Triggered on: boarding, alighting, delays, geofence breach.

  aiService.js
    - HTTP client (Axios) that wraps calls to the Python AI service.
    - Methods: encodeface(imageBuffer), compareFaces(), identifyFace().
    - Used by scan.controller.js and student.controller.js.

  absenteeService.js
    - Runs scheduled checks (via node-cron) at trip end time.
    - Detects students who never boarded.
    - Fires "absent" notifications to parents.

  schedulerService.js
    - Cron job manager.
    - Schedules: absentee checks, trip status resets, daily summaries.

  storageService.js
    - Handles file upload to local disk (via Multer).
    - Stores student and driver face photos in /uploads/ directory.

  SOCKETS (src/sockets/)
  ──────────────────────
  index.js  (Socket.IO Handler)
    - Manages all real-time WebSocket events.
    - Events emitted TO clients:
        "locationUpdate"   - { busId, lat, lng, speed, studentsOnBoard }
        "studentBoarded"   - { studentId, busId, timestamp }
        "studentAlighted"  - { studentId, busId, timestamp }
        "tripStarted"      - { busId, routeId, driverId }
        "tripEnded"        - { busId, summary }
        "alert"            - { type, message, severity }
    - Events received FROM driver app:
        "driverLocation"   - Broadcasts to parent/admin rooms.
        "scanResult"       - Forwards boarding/alighting events.

  UTILITIES (src/utils/)
  ──────────────────────
  Various helper utilities for token generation, distance calculation,
  response formatting, and date manipulation.

  8. MODULE 5: AI SERVICE (FACE RECOGNITION / ML)
================================================================================

  Location:  e:\SafeRide-master\ai-service\
  Port:      8001
  Language:  Python 3.9+
  Run:       uvicorn app.main:app --reload --port 8001

  MAIN FILE
  ─────────
  app/main.py
    - Complete FastAPI application. Single file containing all code.
    - Framework: FastAPI (async, OpenAPI auto-docs at /docs).

  HOW THE FACE RECOGNITION WORKS
  ───────────────────────────────
  Library:  face_recognition (by Adam Geitgey)
  Algo:     Uses dlib's 68-point facial landmark detection.
            Produces a 128-dimensional embedding vector per face.
            These embeddings are compared using Euclidean distance.
  Model:    HOG (Histogram of Oriented Gradients) — fast, CPU-based.
            Alternative: CNN (more accurate, needs GPU).

  ENDPOINTS
  ─────────
  GET  /health
    - Health check. Returns service status and capabilities.

  POST /api/v1/face/encode (multipart file upload)
    - Accepts: image file (JPEG/PNG).
    - Process: detect face → extract 128D embedding vector.
    - Returns: { success, face_encoding: float[128], faces_detected }.
    - Used when: admin uploads a new student or driver photo.

  POST /api/v1/face/encode-base64
    - Same as above but accepts base64-encoded image string.
    - Used when: driver app sends live camera frame as base64.

  POST /api/v1/face/compare
    - Accepts: two 128D encoding arrays + tolerance (default 0.6).
    - Process: calculates Euclidean distance between vectors.
    - Returns: { is_match: bool, distance: float, confidence: float }.

  POST /api/v1/face/identify
    - Accepts: one probe encoding + list of {id, encoding} objects.
    - Process: compares probe against all known encodings, finds closest.
    - Returns: { found: bool, matched_id, distance, confidence }.
    - Used when: driver scans a student's face to verify identity.

  FACE RECOGNITION PIPELINE (end-to-end)
  ───────────────────────────────────────
  1. Admin uploads student photo via Admin Dashboard.
  2. API Gateway (student.controller.js) forwards to /face/encode.
  3. AI Service returns 128D float array.
  4. API Gateway stores the array in Student.faceEncoding in MongoDB.

  VERIFICATION AT BUS (scan):
  1. Driver opens ScanScreen in driver app.
  2. Camera captures face → sends base64 to API Gateway.
  3. API Gateway calls /face/encode-base64 on AI service → gets probe encoding.
  4. API Gateway fetches all students' stored encodings for that bus.
  5. API Gateway calls /face/identify with probe + known encodings.
  6. AI Service returns matched studentId (or not found).
  7. API Gateway fires Socket.IO "studentBoarded" → real-time parent alert.

  ML METRICS
  ──────────
  Encoding dimensions:  128 floats per face
  Match tolerance:      0.6 Euclidean distance (configurable)
  Confidence formula:   (1 - distance/tolerance) × 100%
  Accuracy:             ~99.38% on LFW benchmark (dlib baseline)

================================================================================
  11. KEY TECHNOLOGIES DEEP-DIVE
================================================================================

  GPS & MAP IMPLEMENTATION
  ─────────────────────────
  Library:     Leaflet 1.9.4 + React Leaflet 4.2.1
  Map Tiles:   OpenStreetMap (https://tile.openstreetmap.org/{z}/{x}/{y}.png)
               Free, no API key required.
  Components:  <MapContainer>, <TileLayer>, <Marker>, <Popup> from react-leaflet.
  GPS Source:  Driver's phone via Expo Location API.
               navigator.geolocation on web (browser GPS).
  Update Rate: Every 3 seconds (driver app → Socket.IO → API Gateway → clients).
  Used in:     parent-portal/src/pages/LiveTracking.jsx
               admin-dashboard/src/pages/LiveTracking.jsx
               driver-app/src/screens/HomeScreen.jsx

  REAL-TIME COMMUNICATION
  ───────────────────────
  Protocol:    WebSocket (Socket.IO over ws://)
  Server:      socket.io in api-gateway/src/sockets/index.js
  Clients:     socket.io-client in parent portal, admin dashboard, driver app.
  Rooms:       Each bus has its own Socket.IO room (busId).
               Parents join the room for their child's bus automatically.
               Admin joins all rooms.
  Key Events:
    driverLocation   → GPS lat/lng from driver app to server
    locationUpdate   → Server broadcasts GPS to parent/admin
    studentBoarded   → AI scan verified boarding
    studentAlighted  → AI scan verified exit
    alert            → System-level alerts to parents

  FACE RECOGNITION / ML
  ─────────────────────
  Library:       face_recognition (Python) — built on dlib
  Algorithm:     Deep metric learning (ResNet-based face embedding)
  Vector Size:   128 dimensions per face
  Distance:      Euclidean: sqrt(sum((v1[i]-v2[i])^2 for i in 0..127))
  Threshold:     0.6 = match (tunable per installation)
  Processing:    CPU-only by default (HOG model). GPU possible (CNN model).
  Storage:       128D float array stored per student in MongoDB.
  NOT used:      No Keras, TensorFlow or PyTorch directly. The face_recognition
                 library wraps dlib's pre-trained ResNet model internally.

  AUTHENTICATION & SECURITY
  ──────────────────────────
  JWT:           jsonwebtoken — HS256 signed tokens
  Roles:         "parent", "admin", "driver"
  Hashing:       bcryptjs with 10 salt rounds
  Headers:       Helmet.js sets Content-Security-Policy, X-Frame-Options, etc.
  Validation:    Zod schemas on all request bodies
  CORS:          Configured per-origin in Express and FastAPI

  13. DATA FLOW SUMMARY
================================================================================

  PARENT TRACKING A BUS
  ──────────────────────
  Driver App (GPS) → Socket.IO "driverLocation"
    → API Gateway (sockets/index.js) → stores in BusLocation collection
    → broadcasts "locationUpdate" to bus room
    → Parent Portal (LiveTracking.jsx) receives event
    → React state update → Leaflet Marker moves on map

  STUDENT BOARDING VERIFICATION (FACE SCAN)
  ──────────────────────────────────────────
  1. Driver App (ScanScreen) captures photo → base64
  2. POST /api/scan/verify → API Gateway (scan.controller.js)
  3. API Gateway → POST /api/v1/face/encode-base64 → AI Service
  4. AI Service returns 128D probe encoding
  5. API Gateway fetches all student encodings for bus from MongoDB
  6. API Gateway → POST /api/v1/face/identify → AI Service
  7. AI Service returns matched studentId + confidence %
  8. API Gateway marks Student.boardingHistory (MongoDB)
  9. Emits "studentBoarded" via Socket.IO
  10. Parent Portal receives event → shows "Your child has boarded Bus #3"
  11. notificationService.js → sends SMS (Twilio) + Email (Nodemailer)

  PARENT REGISTRATION & CHILD LINK
  ──────────────────────────────────
  1. Parent registers on Parent Portal (Register.jsx)
  2. POST /api/auth/register → API Gateway
  3. User document created in MongoDB (role: "parent")
  4. Admin adds student in Admin Dashboard → selects parent email
  5. Student.parentRef linked to parent User document
  6. Parent can now see child in Home.jsx and track on LiveTracking.jsx
